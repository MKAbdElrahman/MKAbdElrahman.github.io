Event-Driven Architecture (EDA)

EDA is an architectual style where applications communicate by asynchronously responding to events. Events represent something happened in the system, such as a user placing an order, a product being updated, or a payment being completed.

Types of Events:

  • Initiating Events: These are events that originate from outside the system or represent the starting point of a workflow. Examples include a user placing an order, a sensor detecting a change, or an external system sending data.
  • Derived Events: These events are generated within the system as a consequence of processing an initiating event or another derived event. They represent a change in state or the completion of a specific step in the workflow. For instance, an “Order Placed” event could trigger a derived event like “Inventory Updated” or “Payment Processing Initiated.”

Components:

  • Event Processors : Services that publish or subscribe to events and take some action.

  • Event Channels: Act as intermediaries for event transmission. These channels, often called message queues or event brokers, ensure reliable and ordered delivery of events to interested consumers.

Key Points:

  • Asynchronous Communication: Event-driven systems are asynchronous, meaning the producer doesn’t wait for a response from the consumer after publishing an event. This allows services to be loosely coupled and independently scalable.
  • Loose Coupling: Services are not tightly bound to each other. They only need to know the format of the events they produce or consume, not the specifics of how other services process them.
  • Scalability: Event-driven architectures can be easily scaled by adding more producers or consumers independently.
